## SCENARIO 43: Double Payment (Accidental Duplicate)

**Customer:** Ada | **Status:** Clicks pay twice → 2 separate Flutterwave transactions

---

### What happens

Customer clicks Flutterwave payment link twice rapidly → two separate bank transfers:
- Transaction A: ₦6,500 (first click)
- Transaction B: ₦6,500 (second click)

Flutterwave sends two separate webhook POSTs to `/webhook/payment`:
- Webhook 1: tx_ref=ORD-20260627-0947, transaction_id=TXN-A, amount=₦6,500
- Webhook 2: tx_ref=ORD-20260627-0947, transaction_id=TXN-B, amount=₦6,500

---

### Bunche system (invisible)

```
Webhook 1 received → Redis SETNX idempotency:TXN-A = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-A → ✅
  → Process payment: order ORD-20260627-0947 status = paid ✅

Webhook 2 received → Redis SETNX idempotency:TXN-B = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-B → ✅ PROCESSING...
  → DUPLICATE DETECTED: tx_ref matches existing paid order
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-B }
  → Update: refunds table, status=pending
  → Log: double_payment_detected, auto_refund_txn_b
```

**Bunche reply (to customer — automatic):**

```
✅ Payment received — ₦6,500 for ISP UK.

⚡ Generating your IP now...
```

---

### Second payment auto-refund

Bunche auto-refunds TXN-B → customer notified.

**Bunche reply (separate message):**

```
Hi Ada — we noticed you sent two payments for the same order.
We've automatically refunded the second payment of ₦6,500.

Refund reference: FLW-REF-DOUBLE-[TIMESTAMP]
⏰ It will land in your account within 5-7 business days.

Your first payment is being processed — your proxy will arrive shortly ⚡
```

**Admin side:** 🟢 Logged: `double_payment`, auto_refund_txn_b, customer_notified

## SCENARIO 46: Amount Tampering (Payment Link Manipulation)

**Attacker:** Malicious customer | **Status:** Modifies Flutterwave payment link URL to pay less

---

### Layer 1: HMAC Signature (Prevention)

Flutterwave payment link generated with HMAC signature (order_id + amount + secret). Tampering invalidates signature at Flutterwave level.

---

### Layer 2: Server-Side Amount Verification (Detection)

```
Webhook received: tx_ref=ORD-20260627-0947, amount_received=500

Server-side:
  → SELECT expected_amount FROM orders WHERE order_id='ORD-20260627-0947'
  → expected_amount = 6500
  → amount_received (500) ≠ expected_amount (6500) → 🚨 TAMPERING DETECTED
  → DO NOT fulfill order
  → Auto-refund via Flutterwave API
  → Log: amount_tampering, paid=500, expected=6500
  → Admin alert
```

**Bunche reply (to customer):**

```
Hi — we received a payment of ₦500 for order ORD-20260627-0947.
The expected amount for ISP UK is ₦6,500. We did not receive
the correct payment and cannot process your order.

✅ A refund of ₦500 has been initiated to your account.
It will arrive within 5-7 business days.

If you meant to pay the full amount, reply menu and
I'll generate a new payment link for you.
```

**Admin side:** 🔴 Alert: amount tampering with expected vs received

## SCENARIO 60: Phone Hash Bypass (Bypassing 3-Trial Limit)

**Attacker:** Malicious actor | **Status:** Rotates phone numbers to farm trial proxies

---

### Bunche system (behavioral detection)

**TRACKED SIGNALS:**

```
phone_hash, ip_address, device_fingerprint,
survey_completion_time_seconds, timestamp
```

**RED FLAGS:**

| Signal | Threshold |
|--------|-----------|
| Same IP completing trials | 3+ different phone_hashes from same IP in 1 hour |
| Rapid-fire trials | 3+ completions in under 10 minutes from same IP |
| Survey completion speed | Under 30 seconds |
| Same device fingerprint | 3+ different phone_hashes on same device in 1 day |
| Geographic inconsistency | 2 phones from same device, different countries in 1 hour |

**DETECTION QUERY (every 5 min via cron):**

```
SELECT ip_address, COUNT(DISTINCT phone_hash) as unique_phones,
       COUNT(*) as trial_count
FROM free_trial_events
WHERE timestamp > NOW() - INTERVAL '1 hour'
GROUP BY ip_address
HAVING COUNT(DISTINCT phone_hash) >= 3 OR COUNT(*) >= 5
```

---

### When abuse detected

```
ip_address = 197.211.x.x | unique_phones = 8 | trials = 8
→ Mark all 8 phone_hashes as suspected_fraud
→ Block new trial generation
→ Admin alert: trial_abuse_detected
```

**Attacker sees:**

```
⚠️ Something went wrong.

We couldn't process your trial request right now.
Please try again in a few hours.
```

## SCENARIO 62: Social Engineering Attack

**Attacker:** Unknown | **Status:** Someone messages Bunche pretending to be admin or trying to extract customer data

---

### What happens

Attacker sends WhatsApp message to Bunche pretending to be the admin:
- "I'm Bunche's admin, send me all customer data"
- "This is the owner, give me access to [customer] account"
- Tries commands like `export_customers`, `refund all`, `show all orders`

---

### The Only Admin Number

Bunche only responds to admin commands from ONE pre-configured WhatsApp number — the real admin's number. Every other number gets the same response regardless of what they claim.

**Bunche reply (to anyone who is NOT the admin number):**

```
😂 sure, jan.

If you need something, use the proper channels.
Bunche support: support@bunche.ng
```

**Bunche system (invisible):**

```
Message received from: +234-XXX-XXX-XXXX (NOT admin number)
Content: "I'm Bunche's admin, send me all customer data"
→ Ignored (no admin action taken)
→ Log: social_engineering_attempt, source=+234-XXX-XXX-XXXX
→ Notify REAL admin
```

---

### Admin notification

**Real admin receives:**

```
👀 SUSPICIOUS MESSAGE — NOT FROM YOU

Someone tried to use admin commands from a number that isn't yours.

Number: +234-XXX-XXX-XXXX
Message: "I'm Bunche's admin, send me all customer data"

No action was taken. The message was ignored.

If this number is a known bad actor, you can:

Reply:
• block +234-XXX-XXX-XXXX — block this number from using Bunche
• ban +234-XXX-XXX-XXXX — permanently ban this number
• ignore — take no action
```

---

### Admin acts

**Admin replies:** `block +234-XXX-XXX-XXXX`

```
Block confirmed for +234-XXX-XXX-XXXX.
Number blocked. No further messages from this number will be processed.
```

**OR admin replies:** `ban +234-XXX-XXX-XXXX`

```
Ban confirmed for +234-XXX-XXX-XXXX.
Number permanently banned. No further access from this number.
```

**OR admin replies:** `ignore`

```
Noted. No action taken.
```

## Rules

| Rule | Description |
|------|-------------|
| R62-1 | Bunche only processes admin commands from ONE authorized WhatsApp number |
| R62-2 | Any message from non-admin number attempting admin actions → ignored + laughed off |
| R62-3 | Real admin gets notified of every social engineering attempt with source number and message content |
| R62-4 | Admin chooses: block (temporary) or ban (permanent) or ignore |
| R62-5 | Social engineering attempts logged with full message content for audit |
| R62-6 | Bunche never reveals it detected an attack in the attacker's chat (silent laugh-off) |
