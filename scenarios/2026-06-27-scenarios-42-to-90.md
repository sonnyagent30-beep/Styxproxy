## SCENARIO 43: Double Payment (Accidental Duplicate)

**Customer:** Ada | **Status:** Clicks pay twice → 2 separate Flutterwave transactions

---

### What happens

Customer clicks Flutterwave payment link twice rapidly → two separate bank transfers:
- Transaction A: ₦6,500 (first click)
- Transaction B: ₦6,500 (second click)

Flutterwave sends two separate webhook POSTs to `/webhook/payment`:
- Webhook 1: tx_ref=ORD-20260627-0947, transaction_id=TXN-A, amount=₦6,500
- Webhook 2: tx_ref=ORD-20260627-0947, transaction_id=TXN-B, amount=₦6,500

---

### Bunche system (invisible)

```
Webhook 1 received → Redis SETNX idempotency:TXN-A = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-A → ✅
  → Process payment: order ORD-20260627-0947 status = paid ✅

Webhook 2 received → Redis SETNX idempotency:TXN-B = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-B → ✅ PROCESSING...
  → DUPLICATE DETECTED: tx_ref matches existing paid order
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-B }
  → Update: refunds table, status=pending
  → Log: double_payment_detected, auto_refund_initiated, TXN-B
```

**Bunche reply (to customer — automatic):**

```
✅ Payment received — ₦6,500 for ISP UK.

⚡ Generating your IP now...
```

---

### Second payment auto-refund

Bunche's Flutterwave integration auto-initiates refund for TXN-B → Flutterwave processes → customer receives ₦6,500 back within 5-7 business days.

**Bunche reply (separate message):**

```
Hi Ada — we noticed you sent two payments for the same order.
We've automatically refunded the second payment of ₦6,500.

Refund reference: FLW-REF-DOUBLE-[TIMESTAMP]
⏰ It will land in your account within 5-7 business days.

Your first payment is being processed — your proxy will arrive shortly ⚡
```

**Admin side:** 🟢 Logged: `double_payment`, auto_refund_txn_b, customer_notified

---

## CRITICAL SECURITY NOTE

Flutterwave tx_ref is customer-controlled input. The tx_ref format `ORD-YYYYMMDD-XXXX` is predictable. In Scenario 46 (amount tampering), we address this. For double-payment, the idempotency key is the Flutterwave `transaction_id` (Flutterwave-controlled), NOT `tx_ref` — so this scenario is protected.

---

## Rules

| Rule | Description |
|------|-------------|
| R43-1 | Idempotency key = Flutterwave transaction_id (not tx_ref) |
| R43-2 | Duplicate transaction detected → auto-refund second payment immediately |
| R43-3 | Customer notified of double payment + refund reference |
| R43-4 | First payment always processed normally |
| R43-5 | All double-payment events logged with both transaction IDs |

## SCENARIO 45: Chargeback / Dispute (Bank Reversal)

**Customer:** Emeka | **Status:** Paid ₦6,500 → IP delivered → works 5 days → bank initiates chargeback

---

### What happens

1. Emeka pays ₦6,500 → IP delivered → proxy works fine
2. Day 5: Bank (via Flutterwave) sends chargeback notification — Emeka disputed the charge
3. Bunche receives Flutterwave `chargeback.received` webhook
4. ₦6,500 deducted from Bunche's Flutterwave balance
5. IP is still active (Emeka has been using it for 5 days, 22 days remaining)

---

### Bunche system (invisible)

```
Flutterwave chargeback webhook received:
  → order ORD-20260627-0947: status = chargeback_received
  → Log: chargeback, customer=Emeka, amount=6500, days_active=5

INVESTIGATION PHASE:
1. Check proxy health at time of chargeback:
   → curl test to IP → RESPONDING ✅ (IP was live when chargeback came in)
   → Days since delivery: 5
   → Days remaining on proxy: 22
   → IP usage logs: active daily usage for 5 days

2. Check customer history:
   → First-time customer or repeat?
   → Ban claims filed? → None
   → Other disputes? → None

DECISION MATRIX:
  • IP was LIVE at chargeback time → partial refund offer (22 days remaining)
  • IP was DEAD at chargeback time → full refund (customer gets nothing)
  • Repeat customer with history of disputes → no refund, flag account
```

**Bunche reply (IP was live — partial refund offer):**

```
Hi Emeka — we received a chargeback notification from your bank.

Here's what our records show:
• Your ISP UK proxy was active and working for 5 days
• 22 days remaining on your subscription
• We haven't received any complaint about your IP

Since the proxy was working throughout this period, we're offering:

✅ PARTIAL REFUND: ₦4,733 (₦6,500 × 22/30 days remaining)
   → Refund issued to your bank account within 5-7 business days

Your proxy stays active until [original expiry date].

If you'd prefer a full refund instead (no proxy access going forward),
reply full refund and we'll process it.
```

**Bunche reply (IP was dead at chargeback time — full refund):**

```
Hi Emeka — we received a chargeback notification from your bank.

Our records show your proxy was not working at the time of the dispute.
We've processed a full refund of ₦6,500 to your bank account.

We're sorry the experience wasn't what you expected.
If you'd like to try again with a different product, reply menu.
```

**Bunche reply (repeat customer, no valid claim — deny):**

```
Hi Emeka — we've reviewed the chargeback for order ORD-20260627-0947.

Our records confirm your proxy was active and working throughout
the subscription period. Your bank dispute has been reviewed
and we are disputing it with Flutterwave.

No refund will be issued. Your proxy remains active.
If you have concerns, reply with details.
```

**Admin side:** 🔴 Alert:
```

Amount: ₦6,500

```

## Rules

| Rule | Description |
|------|-------------|
| R45-1 | Chargeback received → order status = chargeback_received, IP stays active during investigation |
| R45-2 | Investigation checks: IP health at chargeback time, customer history, usage logs |
| R45-3 | IP was LIVE at chargeback → offer partial refund proportional to unused days |
| R45-4 | IP was DEAD at chargeback → full refund |
| R45-5 | Repeat customer with chargeback history → dispute with Flutterwave, deny refund |
| R45-6 | All chargeback decisions logged with evidence (IP test results, usage logs) |
