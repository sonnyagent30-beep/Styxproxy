## SCENARIO 43: Double Payment (Accidental Duplicate)

**Customer:** Ada | **Status:** Clicks pay twice → 2 separate Flutterwave transactions

---

### What happens

Customer clicks Flutterwave payment link twice rapidly → two separate bank transfers.

Flutterwave sends two separate webhook POSTs:
- Webhook 1: tx_ref=ORD-20260627-0947, transaction_id=TXN-A, amount=₦6,500 → processed ✅
- Webhook 2: tx_ref=ORD-20260627-0947, transaction_id=TXN-B, amount=₦6,500 → duplicate detected

### Bunche system (invisible)

```
Webhook 2 received → DUPLICATE DETECTED
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-B }
  → Log: double_payment_detected, auto_refund_txn_b
```

**Bunche reply:**

```
✅ Payment received — ₦6,500 for ISP UK.
⚡ Generating your IP now...
```

**Bunche reply (second message):**

```
Hi Ada — we noticed you sent two payments for the same order.
We've automatically refunded the second payment of ₦6,500.

Refund reference: FLW-REF-DOUBLE-[TIMESTAMP]
⏰ It will land in your account within 5-7 business days.
```

**Admin side:** 🟢 Logged: `double_payment`, auto_refund_txn_b

## SCENARIO 46: Amount Tampering (Payment Link Manipulation)

**Attacker:** Modifies Flutterwave payment link URL to pay less

---

### Layer 1: HMAC Signature (Prevention)

Payment link generated with HMAC signature (order_id + amount + secret).
Tampering invalidates signature at Flutterwave level → transaction flagged.

---

### Layer 2: Server-Side Verification (Detection)

```
Webhook: tx_ref=ORD-20260627-0947, amount_received=500
  → SELECT expected_amount FROM orders WHERE order_id='ORD-20260627-0947'
  → expected_amount = 6500
  → amount_received (500) ≠ expected_amount (6500) → 🚨 TAMPERING
  → DO NOT fulfill → Auto-refund → Admin alert
```

**Bunche reply:**

```
Hi — we received a payment of ₦500 for order ORD-20260627-0947.
The expected amount for ISP UK is ₦6,500. We did not receive
the correct payment and cannot process your order.

✅ A refund of ₦500 has been initiated to your account.
It will arrive within 5-7 business days.

If you meant to pay the full amount, reply menu and
I'll generate a new payment link for you.
```

**Admin side:** 🔴 Alert: amount tampering detected

## SCENARIO 60: Phone Hash Bypass (Bypassing 3-Trial Limit)

**Attacker:** Rotates phone numbers to farm trial proxies

---

### Bunche system (behavioral detection)

**TRACKED:** phone_hash, ip_address, device_fingerprint, survey_completion_time

**RED FLAGS:**

| Signal | Threshold |
|--------|-----------|
| Same IP completing trials | 3+ different phone_hashes from same IP in 1 hour |
| Rapid-fire trials | 3+ completions in under 10 minutes from same IP |
| Survey completion | Under 30 seconds |
| Same device fingerprint | 3+ different phone_hashes on same device in 1 day |
| Geographic inconsistency | 2 phones from same device, different countries in 1 hour |

**DETECTION QUERY (every 5 min):**

```
SELECT ip_address, COUNT(DISTINCT phone_hash) as unique_phones
FROM free_trial_events
WHERE timestamp > NOW() - INTERVAL '1 hour'
GROUP BY ip_address
HAVING COUNT(DISTINCT phone_hash) >= 3 OR COUNT(*) >= 5
```

---

### When abuse detected

```
ip=197.211.x.x | unique_phones=8 | trials=8
→ Mark all 8 phone_hashes as suspected_fraud
→ Block new trial generation
→ Admin alert: trial_abuse_detected
```

**Attacker sees:**

```
⚠️ Something went wrong.
We couldn't process your trial request right now.
Please try again in a few hours.
```

## SCENARIO 64: Impersonation (Refund Request on Working IP)

**Attacker/Customer:** Claims they never ordered — wants refund on a working IP

---

### What happens

Someone messages Bunche:
- Option A: Real customer lying to get free proxy + refund
- Option B: Someone trying to social-engineer a refund for someone else's order
- Option C: Someone who genuinely forgot they ordered

---

### Step 1: Identity Verification (BOTH name + PIN required)

**Bunche reply:**

```
Hi — before we can look into any order, I need to verify your identity.

Please reply with:
• Your referral name (e.g. Dan, Ada, Tunde22)
• Your 4-digit PIN

⚠️ Wrong PIN 3 times = locked + admin alert.
```

**If they can't provide both:** No further help. Anti-social-engineering gate.

---

### Step 2: Check Order + Proxy Status

After identity confirmed (name + PIN verified):

```
SELECT * FROM orders WHERE customer_id = ? AND status = 'fulfilled'
Order: ORD-20260627-0947 | Product: ISP UK | Delivered: 2026-06-27 10:00

IP health check NOW (2026-06-27 14:00 — 4 hours later):
  → curl test to IP → RESPONDING ✅ (proxy is live)

Days since delivery: 0 (same day)
IP status: WORKING
Hours elapsed since delivery: 4
```

### Anti-Leak Rule

Bunche NEVER reveals order details (product, date, amount, IP) to someone who can't verify identity. Even if someone says "I ordered yesterday and the proxy is slow" — Bunche doesn't confirm whether an order exists until name + PIN verify.
