## SCENARIO 43: Double Payment (Accidental Duplicate)

**Customer:** Ada | **Status:** Clicks pay twice → 2 separate Flutterwave transactions

---

### What happens

Customer clicks Flutterwave payment link twice rapidly → two separate bank transfers:
- Transaction A: ₦6,500 (first click)
- Transaction B: ₦6,500 (second click)

Flutterwave sends two separate webhook POSTs to `/webhook/payment`:
- Webhook 1: tx_ref=ORD-20260627-0947, transaction_id=TXN-A, amount=₦6,500
- Webhook 2: tx_ref=ORD-20260627-0947, transaction_id=TXN-B, amount=₦6,500

---

### Bunche system (invisible)

```
Webhook 1 received → Redis SETNX idempotency:TXN-A = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-A → ✅
  → Process payment: order ORD-20260627-0947 status = paid ✅

Webhook 2 received → Redis SETNX idempotency:TXN-B = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-B → ✅ PROCESSING...
  → DUPLICATE DETECTED: tx_ref matches existing paid order
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-B }
  → Update: refunds table, status=pending
  → Log: double_payment_detected, auto_refund_initiated, TXN-B
```

**Bunche reply (to customer — automatic):**

```
✅ Payment received — ₦6,500 for ISP UK.

⚡ Generating your IP now...
```

---

### Second payment auto-refund

Bunche's Flutterwave integration auto-initiates refund for TXN-B → Flutterwave processes → customer receives ₦6,500 back within 5-7 business days.

**Bunche reply (separate message):**

```
Hi Ada — we noticed you sent two payments for the same order.
We've automatically refunded the second payment of ₦6,500.

Refund reference: FLW-REF-DOUBLE-[TIMESTAMP]
⏰ It will land in your account within 5-7 business days.

Your first payment is being processed — your proxy will arrive shortly ⚡
```

**Admin side:** 🟢 Logged: `double_payment`, auto_refund_txn_b, customer_notified

## SCENARIO 46: Amount Tampering (Payment Link Manipulation)

**Attacker:** Malicious customer | **Status:** Modifies Flutterwave payment link URL to pay less

---

### What happens

1. Bunche generates payment link with HMAC signature
2. Attacker modifies URL to change amount before paying
3. Payment arrives at lower amount than expected

---

### Layer 1: HMAC Signature (Prevention)

Flutterwave payment link generated with HMAC signature (order_id + amount + secret).

**If HMAC check fails at Flutterwave level:**
→ Transaction flagged before reaching Bunche's webhook

---

### Layer 2: Server-Side Amount Verification (Detection)

Webhook handler always verifies:

```
Webhook received: tx_ref=ORD-20260627-0947, amount_received=500

Server-side check:
  → SELECT expected_amount FROM orders WHERE order_id='ORD-20260627-0947'
  → expected_amount = 6500
  → amount_received (500) ≠ expected_amount (6500) → 🚨 TAMPERING DETECTED
```

**Bunche system (invisible):**

```
Webhook received → amount mismatch detected
  → DO NOT fulfill order
  → Order status = tampered_payment_rejected
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-TAMPERED }
  → Log: amount_tampering, order=ORD-20260627-0947, paid=500, expected=6500
  → Admin alert
```

**Bunche reply (to customer):**

```
Hi — we received a payment of ₦500 for order ORD-20260627-0947.

The expected amount for ISP UK is ₦6,500. We did not receive
the correct payment and cannot process your order.

✅ A refund of ₦500 has been initiated to your account.
It will arrive within 5-7 business days.

If you meant to pay the full amount, reply menu and
I'll generate a new payment link for you.
```

**Admin side:** 🔴 Alert:
```
🚨 AMOUNT TAMPERING DETECTED

Order: ORD-20260627-0947
Expected: ₦6,500 | Received: ₦500
Customer: (a3f2b9...)
Action: Refund initiated. Order not fulfilled.
```

## SCENARIO 60: Phone Hash Bypass (Bypassing 3-Trial Limit)

**Attacker:** Malicious actor | **Status:** Uses multiple phone numbers to bypass 3-trials/day limit

---

### What happens

Attacker creates multiple WhatsApp accounts / SIMs → rotates phone numbers to get fresh trials each time:
- Trial #1: +234-801-111-1111 → completes survey → gets trial proxy
- Trial #2: +234-801-111-1112 → completes survey → gets trial proxy
- Trial #3: +234-801-111-1113 → completes survey → gets trial proxy
- Trial #4: +234-801-111-1114 → tries again → phone_hash check passes (new number)

Attacker cycles through 10+ numbers to farm trial proxies.

---

### Bunche system (behavioral detection)

Phone_hash limit is still the first line of defense — but Bunche ALSO tracks behavioral signals:

**TRACKED SIGNALS:**

```
Trial event logged:
  phone_hash = sha256("+2348011111111")
  ip_address = 197.211.x.x
  device_fingerprint = [browser, OS, WhatsApp version]
  trial_number = 1
  timestamp = 2026-06-27 10:00:00
  survey_completion_time_seconds = 47  ← fast = suspicious
```

**BEHAVIORAL RED FLAGS (any ONE triggers investigation):**

| Signal | Threshold | Why |
|--------|-----------|-----|
| Same IP completing trials | 3+ different phone_hashes from same IP in 1 hour | Phone rotation pattern |
| Rapid-fire trials | 3+ trial completions in under 10 minutes from same IP | Automated/scripted |
| Survey completion speed | Under 30 seconds | Fake survey or bot |
| Same device fingerprint | 3+ different phone_hashes on same device in 1 day | SIM farm / emulator |
| Geographic inconsistency | 2 phone_numbers from same device but different countries within 1 hour | Impossible travel |

**DETECTION QUERY (run every 5 minutes via cron):**

```
SELECT ip_address, COUNT(DISTINCT phone_hash) as unique_phones,
       COUNT(*) as trial_count, MIN(timestamp), MAX(timestamp)
FROM free_trial_events
WHERE timestamp > NOW() - INTERVAL '1 hour'
GROUP BY ip_address
HAVING COUNT(DISTINCT phone_hash) >= 3
   OR COUNT(*) >= 5
   OR SUM(CASE WHEN completion_time_seconds < 30 THEN 1 ELSE 0 END) >= 3
```

---

### When behavioral abuse detected

**Bunche system (invisible):**

```
Behavioral abuse pattern detected:
  ip_address = 197.211.x.x
  unique_phones = 8
  trial_count = 8
  time_window = 47 minutes
  → Mark all 8 phone_hashes as suspected_fraud
  → Log: trial_abuse_detected, ip=197.211.x.x, phones=8
  → Block new trial generation for these phone_hashes
  → Admin alert
```

**Admin side:** 🚨 Alert:
```
🚨 TRIAL ABUSE — PHONE ROTATION DETECTED

IP: 197.211.x.x
Unique phone numbers: 8
Trials in last hour: 8
Pattern: Phone hash bypass (behavioral detection)

Action taken: All 8 numbers flagged. New trials blocked.
Review: https://bunche.ng/admin/trials?flag=abuse
```

---

### What happens to the attacker

New trial requests from any of the flagged phone_hashes:

**Bunche reply:**

```
⚠️ Something went wrong.

We couldn't process your trial request right now.
Please try again in a few hours.

If this persists, contact admin@bunche.ng.
```

(No mention of abuse detection — customer doesn't know why)

---

### Legitimate multiple-trial users

A real customer who genuinely uses 2-3 devices with different numbers (e.g., phone + wife's phone) will not trigger the behavioral flags because:
- 2-3 trials from same IP over hours (not minutes) is normal
- Different devices have different fingerprints
- Survey completion time is realistic (not <30 seconds)

The threshold of 3+ phones from same IP in 1 hour is designed to catch automated abuse, not casual sharing.

## Rules

| Rule | Description |
|------|-------------|
| R60-1 | Phone_hash limit = 3 trials/day (first line of defense) |
| R60-2 | Behavioral detection = secondary layer (catches phone rotation) |
| R60-3 | Behavioral signals tracked: IP, device fingerprint, survey completion time |
| R60-4 | 3+ unique phones from same IP in 1 hour → flag as abuse |
| R60-5 | 5+ trials from same IP in 1 hour → flag as abuse |
| R60-6 | Survey completed in under 30 seconds → suspicious (possible bot/fake) |
| R60-7 | Flagged phone_hashes blocked from new trial generation |
| R60-8 | Admin alert sent when abuse pattern detected |
| R60-9 | Customer never told they are flagged or why |
