## SCENARIO 43: Double Payment (Accidental Duplicate)

**Customer:** Ada | **Status:** Clicks pay twice → 2 separate Flutterwave transactions

---

### What happens

Customer clicks Flutterwave payment link twice rapidly → two separate bank transfers.

Flutterwave sends two separate webhook POSTs:
- Webhook 1: tx_ref=ORD-20260627-0947, transaction_id=TXN-A, amount=₦6,500 → processed ✅
- Webhook 2: tx_ref=ORD-20260627-0947, transaction_id=TXN-B, amount=₦6,500 → duplicate detected

### Bunche system (invisible)

```
Webhook 2 received → DUPLICATE DETECTED
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-B }
  → Log: double_payment_detected, auto_refund_txn_b
```

**Bunche reply:**

```
✅ Payment received — ₦6,500 for ISP UK.
⚡ Generating your IP now...
```

**Bunche reply (second message):**

```
Hi Ada — we noticed you sent two payments for the same order.
We've automatically refunded the second payment of ₦6,500.

Refund reference: FLW-REF-DOUBLE-[TIMESTAMP]
⏰ It will land in your account within 5-7 business days.
```

**Admin side:** 🟢 Logged: `double_payment`, auto_refund_txn_b

## SCENARIO 46: Amount Tampering (Payment Link Manipulation)

**Attacker:** Modifies Flutterwave payment link URL to pay less

---

### Layer 1: HMAC Signature (Prevention)

Payment link generated with HMAC signature (order_id + amount + secret).
Tampering invalidates signature at Flutterwave level → transaction flagged.

---

### Layer 2: Server-Side Verification (Detection)

```
Webhook: tx_ref=ORD-20260627-0947, amount_received=500
  → SELECT expected_amount FROM orders WHERE order_id='ORD-20260627-0947'
  → expected_amount = 6500
  → amount_received (500) ≠ expected_amount (6500) → 🚨 TAMPERING
  → DO NOT fulfill → Auto-refund → Admin alert
```

**Bunche reply:**

```
Hi — we received a payment of ₦500 for order ORD-20260627-0947.
The expected amount for ISP UK is ₦6,500. We did not receive
the correct payment and cannot process your order.

✅ A refund of ₦500 has been initiated to your account.
It will arrive within 5-7 business days.

If you meant to pay the full amount, reply menu and
I'll generate a new payment link for you.
```

**Admin side:** 🔴 Alert: amount tampering detected

## SCENARIO 60: Phone Hash Bypass (Bypassing 3-Trial Limit)

**Attacker:** Rotates phone numbers to farm trial proxies

---

### Bunche system (behavioral detection)

**TRACKED:** phone_hash, ip_address, device_fingerprint, survey_completion_time

**RED FLAGS:**

| Signal | Threshold |
|--------|-----------|
| Same IP completing trials | 3+ different phone_hashes from same IP in 1 hour |
| Rapid-fire trials | 3+ completions in under 10 minutes from same IP |
| Survey completion | Under 30 seconds |
| Same device fingerprint | 3+ different phone_hashes on same device in 1 day |
| Geographic inconsistency | 2 phones from same device, different countries in 1 hour |

**DETECTION QUERY (every 5 min):**

```
SELECT ip_address, COUNT(DISTINCT phone_hash) as unique_phones
FROM free_trial_events
WHERE timestamp > NOW() - INTERVAL '1 hour'
GROUP BY ip_address
HAVING COUNT(DISTINCT phone_hash) >= 3 OR COUNT(*) >= 5
```

---

### When abuse detected

```
ip=197.211.x.x | unique_phones=8 | trials=8
→ Mark all 8 phone_hashes as suspected_fraud
→ Block new trial generation
→ Admin alert: trial_abuse_detected
```

**Attacker sees:**

```
⚠️ Something went wrong.
We couldn't process your trial request right now.
Please try again in a few hours.
```

## SCENARIO 77: Trial Then Lie (Claims Paid Without Paying)

**Customer:** Completes free trial → proxy works → messages "I paid" trying to get a second IP without paying

---

### What happens

1. Customer completes free trial → gets trial proxy → proxy works
2. Trial expires (2 hours) OR customer keeps using trial
3. Customer messages: "I paid" or "I paid for the full plan" → trying to get another IP without paying

---

### Layer 1: Webhook Verification

Customer sends "Paid" → Bunche triggers webhook check:

```
Customer: I paid for ISP UK

Bunche system:
  → No order ID provided
  → Check Flutterwave webhooks for any recent successful payment from this phone_hash
  → Webhook search: last 24 hours for phone_hash = this customer
  → RESULT: No payment webhook received ❌

  → ORDER LOOKUP:
    → SELECT * FROM orders WHERE customer_id = ? AND status = 'paid'
    → No paid orders found ❌

    → TRIAL CHECK:
    → SELECT * FROM free_trials WHERE phone_hash = ? AND status = 'active'
    → Trial found: user=trial_a7b9c2, port=8001, expires=2026-06-27 12:00
    → Trial status: ACTIVE or RECENTLY EXPIRED ✅

CONCLUSION: Customer claims to have paid → NO PAYMENT FOUND in Flutterwave
```

---

### Layer 2: Order ID Cross-Check

If customer provides an order ID:

```
Customer: I paid, my order ID is ORD-20260627-1234

Bunche system:
  → SELECT * FROM orders WHERE order_id = 'ORD-20260627-1234'
  → Order NOT FOUND ❌

  OR

  → Order found but:
    status = 'pending_payment' ❌ (not paid)
    OR status = 'trial' ❌ (this is a trial order, not paid)
    OR status = 'refunded' ❌

  → Trial order IDs have format: TRIAL-YYYYMMDD-XXXX
  → Paid order IDs have format: ORD-YYYYMMDD-XXXX
  → If customer says "ORD-..." but it's actually a trial order → 🚨 MISMATCH
```

---

### Bunche reply (no payment found):

```
Hi — I checked our payment records and I don't see any payment
linked to your account.

Our system shows:
• No successful payment from you in the last 24 hours
• An active or recent free trial on your account

If you just paid, it can take up to 5 minutes for the payment
to confirm. If it still doesn't show after 5 minutes, reply menu
and I'll generate a new payment link for you.

If you haven't paid yet and you'd like to:
• ISP UK → ₦6,500/mo
• Residential 5GB → ₦5,000

Reply menu to order.
```

---

### Bunche reply (order ID doesn't match / wrong format):

```
Hi — the order ID you provided (ORD-20260627-XXXX) doesn't match
our records.

Can you double-check the order ID from your payment confirmation?
It should be in your bank app or Flutterwave receipt.

If you don't have an order ID, reply menu to place a new order
and I'll send you a fresh payment link.
```

---

### Bunche reply (suspected fraud — active trial + claims paid):

```
Hi — our records show an active or recent free trial on your account.

You mentioned you paid — our system doesn't show a payment.
Our free trial has already been used on your account.

If you'd like to upgrade to a paid plan:
• ISP UK → ₦6,500/mo
• Residential 5GB → ₦5,000

Reply menu to order.
```

---

If customer continues to insist after being told no payment was found:

**Bunche reply (after 2nd insistence):**

```
Hi — I've already checked our payment records.
```

What if customer DID pay but webhook hasn't fired yet (rare Flutterwave delay)?

```

  → Customer pays at bank
  → Flutterwave processes (can take 1-5 minutes)
  → Webhook hasn't fired yet

Bunche reply (acknowledge possibility):
  → "I don't see your payment yet — give it 5 minutes.

```

---

### Admin alert (repeat liar)

If customer messages multiple times insisting they paid with no payment record:

**Bunche system (invisible):**

```
Repeated "I paid" claims with no payment found:
  → Log: trial_claim_paid_no_payment, phone_hash=...
  → If count >= 3 → flag customer account
```

**Admin side:** 🟡 Logged: `trial_claim_paid_no_payment` — if repeated, consider temporary block
